    java多线程进阶（2）-synchronized关键字

    在上一节中，我们认识了数据同步问题,出现数据同步问题是因为多个线程同时操作共享资源引起的，在JDK1.5之前，要解决这个问题需要用到synchronized关键字，synchronized提供了一种排他机制，该机制只允许同一时间有一个线程操作该共享资源。

    一、什么是synchronized关键字

    synchronized关键字可以实现一个简单的策略来防止线程干扰和内存一致性错误，如果一个对象对多个线程可见，那么对该对象的所有读写或者写都将通过同步的方式来进行，具体如下:

    • synchronized关键字提供了一种锁的机制，能够确保共享变量的互斥访问，从而防止数据不一致问题。
    • synchronized关键字包括monitor enter和monitor exit两个JVM指令，它能够保证在任何时候任何线程执行到monitor enter成功之前都必须从主内存中获取数据，而不是缓存，在monitor exit运行成功之后，共享变量被更新后的值必须刷入主内存(这一点将在后续详细介绍)
    • synchronized的指令遵循java happens-before规则，一个monitor exit指令之前必须要有一个monitor enter

    二、synchronized关键字的用法

    synchronized能够修饰代码块和方法，而不能修饰class和变量。

    2.1 synchronized修饰方法

    synchronized修饰方法的语法非常简单:

    [default|public|protected|private] synchronized [static] type method()

    示例代码如下:

    public synchronized void sync() {
// do something
}

public synchronized static void sync() {
// do something
}

    2.1 synchronized修饰代码块

    private final Object MUTEX = new Object();
public void sync() {
    synchronized (MUTEX) {
        // do something
    }
}

    注意: 很多资料，书籍将synchronized (MUTEX)称为锁，这种说法是不严谨的，准确的说应该是某个线程获取了MUTEX对象的monitor锁

    三、monitor enter和monitor exit指令

    3.1 monitor enter

    每个对象都与一个monitor相关联，一个monitor的lock锁在同一时间只能被一个线程获取，一个线程获取与对象关联的monitor的所有权时，发生如下几件事情:

    • 如果monitor的计数器值为0，则该monitor的lock锁还没有所有者，可以获取，获取后立即将monitor计数器的值置为1。
    • 如果一个拥有该monitor所有权的线程重入，则会导致monitor计数器再次累加。
    • 如果monitor已经被其他线程拥有，则获取该monitor所有权的线程会被阻塞，直到monitor计数器的值变为0，才能再次获取monitor的所有权。

    3.2 monitor exit

    释放monitor的所有权，要释放一个对象关联的monitor的所有权，首先应拥有所有权。释放monitor的过程比较简单，就是将monitor计数器的值置为0。这个过程通俗的将就是解锁。

    四、使用synchronized关键字应注意的问题

    4.1 与monitor关联的对象不能为空

    // 错误示例
private final Object MUTEX = null;
private void sync() {
    synchronized (MUTEX) {
        // do something
    }
}

    每个对象都与一个monitor关联，对象都为null了，更不用谈monitor了

    4.2 synchronized作用域太大

    由于synchronized具有排他性，同一时间只允许一个线程处理同步块的内容，synchronized的作用域越大，处理效率将越低，甚至将丧失多线程的的优势。

    // 错误示例
public static class Task implements Runnable {
    @Override
    public synchronized void run() {
        // do something
    }
}

    上诉代码将完全丧失多线程的优势。

    4.3 不同的monitor企图锁相同的方法

    // 错误示例
public static class Task implements Runnable {
    private final Object MUTEX = new Object();
    @Override
    public void run() {
        // do something
        synchronized(MUTEX) {
            // do something
        }
        // do something
    }
}

public static void main(String args[]) {
    for(int i = 0; i < 5; i++) {
        new Thread(Task::new).start();
    }
}

    上诉代码构造了五个线程，同时也构造了五个Runnable实例，你会发现，synchronized根本互斥不了与之对应的作用域。线程之间monitor lock的争抢只发生在与monito关联的引用上。上面代码每个线程争抢的monitor都关联了不同的引用，即不同的MUTEX实例，因此不可能起到互斥作用。

    4.4 多个锁的交叉导致死锁

    多个锁的交叉使用很容易引起死锁，程序没有任何输出，但就是不工作。

    // 错误示例
private final Object MUTEX_READ = new Object();
private final Object MUTEX_WRITE = new Object();
public void read() {
    synchronized (MUTEX_READ) {
        synchronized (MUTEX_WRITE) {
            // do something
        }
    }
}

public void write() {
    synchronized (MUTEX_WRITE) {
        synchronized (MUTEX_READ) {
            // do something
        }
    }
}
