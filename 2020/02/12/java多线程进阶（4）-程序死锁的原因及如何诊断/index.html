    1、程序死锁的原因

    1.1 交叉锁可能导致程序死锁

    线程A持有R1的锁等待R2的锁，线程B持有R2的锁等待R1的锁，这种情况最容易导致死锁。示例如下：

    public class DeadLock{
    private final Object MUTEX_READ = new Object();
    private final Object MUTEX_WRITE = new Object();

    public void read() {
        synchronized(MUTEX_READ) {
            System.out.println("do something");
            synchronized(MUTEX_WRITE) {
                System.out.println("do something");
            }
        }
    }

    public void write() {
        synchronized(MUTEX_WRITE) {
            System.out.println("do something");
            synchronized(MUTEX_READ) {
                System.out.println("do something");
            }
        }
    }
}

    1.2 死循环引起的死锁

    上诉代码很容易看出程序有死锁的风险，如果使用开源库，API调用层次比较深，那么看代码就不容易发现风险，如下代码使用了JDK中的HashMap，该程序很容出现死循环引起的死锁。

    public class HashMapDeadLock{
    private final HashMap<String, String> map = new HashMap();
    public void add(String key, String value) {
        this.map.put(key, value);
    }
    public static void main(String args[]) {
        final HashMapDeadLock hmdl = new HashMapDeadLock();
        for(int i = 0; i < 2; i++) {
            new Thread(() -> {
                for(int j = 0; j < Integer.MAX_VALUE; j++) {
                    hmdl.add(String.valueOf(j), String.valueOf(j))
                }
            }).start();
        }
    }
}

    HashMap不具备线程安全能力，如果在多线程同时写操作的情况下，不对其进行同步化封装，很容出现死循环引起的死锁，如果想要使用线程安全的map结构，请使用ConcurrentHashMap或者使用Collections.synchronizedMap来代替。

    1.3 内存不足

    当并发请求系统的可用内存时，内存不足，可能会导致死锁。

    1.4 一问一答式的数据交换

    服务端开启某个端口，等待客户端访问，客户端发送请求立即等待接收，由于某种原因，服务的错过了客户端的请求，此时客户端和服务端都等待数据交换，引起死锁。

    1.5 数据库锁

    无论是数据库表级别锁，还是行级别锁，比如某个线程执行for update语句退出了事务，其他线程访问该数据库时都将陷入死锁。

    1.6 文件锁

    同样，某线程获取文件锁后意外退出，其他读取该文件的线程也将会陷入死锁，直到系统释放该文件句柄。

    2、死锁诊断

    在java中我们可以借助诊断工具对其进行诊断。

    2.1 交叉锁引起的死锁

    运行DeadLock代码，程序将进入死锁，打开jstack或者jconsole工具，Jstack-l PID将会直接发现死锁，示例如下：

    一般交叉锁引起的死锁线程都会进入BLOCKED状态，CPU占用率不高。

    2.2 死循环引起的死锁(假死)

    运行HashMapDeadLock程序，也可以使用jstack，jconsole，jvisualvm工具或者jProfiler(收费)来诊断，但不会给出很明显的提示，因为工作线程并没有BLOCKED，而是RUNNABLE状态，且CPU使用率高居不下。
